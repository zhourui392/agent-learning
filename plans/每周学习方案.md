# Agent 8周学习方案（按周落地）

## 使用说明
- 面向对象：有后端经验、熟练使用 Agent 工具的工程师。
- 执行节奏：工作日每天 2 小时，周末每天 3 小时。
- 统一验收：每周末更新 E2E 成功率、P95 延迟、单任务成本、回归通过率。

## W1 架构基线（详细日计划已独立）
- 周目标：完成边界、上下文、状态机、工具契约与最小主链路。
- Day1-7：见 `W1-执行清单.md`。
- 周产物：`docs/architecture/*`、`contracts/*`、最小可运行骨架。

## W2 Workflow 内核（Planner-Executor-Replanner）
- 周目标：把任务流做成可重试、可超时、可幂等、可续跑。
- Day1：定义任务分解协议（PlanStep、依赖关系、完成条件）。
- Day2：实现 Planner（输出结构化计划 + 风险标记）。
- Day3：实现 Executor（串并行执行 + 超时控制 + 取消机制）。
- Day4：实现 Replanner（步骤失败后的重规划策略）。
- Day5：打通状态机联动（失败回滚、快照恢复、幂等键）。
- Day6：补集成测试（超时、重复提交、部分失败、恢复续跑）。
- Day7：复盘与冻结（性能瓶颈、失败样本、W3输入）。
- 周产物：`src/agent/planner.py`、`src/agent/executor.py`、`src/agent/replanner.py`、`tests/integration/workflow_*`。

## W3 RAG 工程化
- 周目标：实现分层检索链路，并形成可回放的评测基线。
- Day1：整理知识源分层（静态文档、动态数据、结构化库）。
- Day2：实现召回层（向量召回 + 关键词召回）。
- Day3：实现重排层（相关性/时效性/权限过滤）。
- Day4：实现上下文压缩（片段裁剪、去重、Token预算）。
- Day5：加入缓存（query缓存、片段缓存、失效策略）。
- Day6：构建离线评测集 v1（问题集、标准答案、评分脚本）。
- Day7：跑基线并沉淀报告（召回率、正确率、成本）。
- 周产物：`src/rag/*`、`eval/datasets/v1/*`、`eval/reports/w3-baseline.md`。

## W4 工具治理
- 周目标：把工具调用变成“可管控的基础设施”。
- Day1：实现工具注册中心（版本、能力、权限标签）。
- Day2：实现调用前校验（Schema、上下文、额度限制）。
- Day3：实现统一错误模型（retryable、degradeable、fatal）。
- Day4：实现熔断/限流/降级策略。
- Day5：接入审计日志（请求、入参、结果、耗时、调用方）。
- Day6：补安全测试（越权调用、注入参数、脏数据）。
- Day7：发布治理清单与工具准入规范。
- 周产物：`src/gateway/*`、`docs/policies/tool-governance.md`、`tests/security/tool_*`。

## W5 评测体系
- 周目标：形成可在 CI 执行的自动化评测与回归门禁。
- Day1：定义指标口径（E2E、步骤成功率、P95、成本）。
- Day2：定义评测样本分层（冒烟、回归、对抗、长尾）。
- Day3：实现评测Runner（批量执行 + 打分 + 归档）。
- Day4：实现结果对比（与上周基线自动Diff）。
- Day5：接入 CI 门禁（未达阈值阻断合并）。
- Day6：补充失败样本回放能力。
- Day7：输出周评测报告与改进列表。
- 周产物：`eval/runner.py`、`eval/config.yaml`、`.github/workflows/eval.yml`（或等价CI脚本）。

## W6 可观测性
- 周目标：实现可定位、可归因、可追责的全链路观测。
- Day1：定义 Trace 模型（trace_id/span_id/session_id/step_id）。
- Day2：埋点关键阶段（规划、检索、工具、汇总、响应）。
- Day3：建立错误分桶（模型类、工具类、数据类、策略类）。
- Day4：建立慢链路分析（TopN 慢步骤、慢工具、慢查询）。
- Day5：构建仪表盘（成功率、延迟、成本、错误趋势）。
- Day6：建立告警规则（SLO 违约、成本突增、错误激增）。
- Day7：演练故障复盘流程（从告警到根因定位）。
- 周产物：`observability/*`、`docs/runbook/incident.md`、`docs/reports/w6-observe.md`。

## W7 多Agent协作
- 周目标：完成多角色协作闭环，并解决冲突与一致性问题。
- Day1：定义角色与职责（Planner/Executor/Auditor）。
- Day2：定义消息协议（输入输出、状态、冲突字段）。
- Day3：实现任务分派与回传机制。
- Day4：实现冲突仲裁（规则优先级、置信度、人工兜底）。
- Day5：实现共享记忆一致性策略（写入顺序、版本冲突处理）。
- Day6：完成至少2个业务流程 Demo。
- Day7：评估单Agent vs 多Agent 成本与收益。
- 周产物：`src/multi_agent/*`、`docs/architecture/multi-agent-protocol.md`、`demo/*`。

## W8 上线与治理
- 周目标：具备可灰度、可回滚、可运维的生产交付能力。
- Day1：制定发布策略（灰度比例、流量切换、回滚条件）。
- Day2：实现 A/B 试验配置与指标看板。
- Day3：建立SLA/SLO监控与告警升级链路。
- Day4：补齐 Runbook（常见故障、人工接管、恢复步骤）。
- Day5：完成压测与容量评估。
- Day6：执行预发布演练（含回滚演练）。
- Day7：正式发布并输出复盘报告。
- 周产物：`docs/runbook/*`、`docs/release/*`、`docs/reports/w8-launch-review.md`。

## 周复盘模板（每周固定）
- 本周目标达成率：`%`
- E2E 成功率：`%`（对比上周）
- P95 延迟：`ms`
- 单任务成本：`$` 或 `tokens`
- Top3 失败原因：`...`
- 下周风险与缓解动作：`...`
