# 索引策略（Index Policy）

## 概述

定义 RAG 系统中文档从原始数据到可检索索引的完整处理流水线，包括切分策略、Embedding 策略、存储方案与更新机制。

---

## 1. 文档切分策略

### 1.1 切分方法

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| 固定窗口切分 | 按 token 数固定长度切分（默认 256 tokens） | 无明显结构的纯文本 |
| 语义段落切分 | 按段落边界（空行 / 标题）切分 | Markdown / 技术文档 |
| 句级切分 | 按句号、换行切分 | 精确事实检索场景 |
| 结构化切分 | 按 JSON key / DB row 切分 | 结构化数据 |

### 1.2 切分参数

```python
CHUNK_CONFIG = {
    "default_chunk_size": 256,       # 默认 chunk 大小（tokens）
    "chunk_overlap": 32,             # 相邻 chunk 重叠量（tokens）
    "max_chunk_size": 512,           # 单 chunk 上限
    "min_chunk_size": 32,            # 低于此长度的 chunk 合并到前一个
    "separator_priority": [          # 切分边界优先级（高→低）
        "\n## ",                     # 二级标题
        "\n### ",                    # 三级标题
        "\n\n",                      # 空行
        "\n",                        # 换行
        ". ",                        # 句号
    ],
}
```

### 1.3 元数据附加

每个 chunk 附加以下元数据，用于后续重排与过滤：

| 字段 | 类型 | 说明 |
|------|------|------|
| `source_id` | string | 原始文档唯一标识 |
| `source_type` | enum | 数据源类型（doc / api / db） |
| `chunk_index` | int | 在原文档中的序号 |
| `total_chunks` | int | 原文档总 chunk 数 |
| `created_at` | datetime | 索引创建时间 |
| `updated_at` | datetime | 最后更新时间 |
| `version` | string | 文档版本号 |
| `access_level` | enum | 访问级别（public / internal / restricted） |
| `token_count` | int | chunk 的 token 数 |

## 2. Embedding 策略

### 2.1 模型选择

| 场景 | 模型 | 维度 | 说明 |
|------|------|------|------|
| 默认 | text-embedding-3-small | 1536 | 性价比最优 |
| 高精度 | text-embedding-3-large | 3072 | 关键场景使用 |
| 离线/私有化 | bge-large-zh-v1.5 | 1024 | 中文场景私有部署 |

### 2.2 Embedding 生成规则

1. 输入文本预处理：去除多余空白、统一标点。
2. 长文本截断：超过模型最大输入长度时截断（而非丢弃）。
3. 批量调用：每批 100 条，降低 API 调用开销。
4. 失败重试：指数退避，最多 3 次。
5. 结果校验：验证返回向量维度与预期一致。

## 3. 存储方案

### 3.1 向量存储

| 组件 | 用途 | 说明 |
|------|------|------|
| FAISS（开发/测试） | 内存向量索引 | 快速原型验证，不持久化 |
| Milvus / Qdrant（生产） | 分布式向量数据库 | 持久化、可扩展、支持过滤 |

### 3.2 关键词索引

| 组件 | 用途 | 说明 |
|------|------|------|
| 内存倒排索引（开发/测试） | BM25 关键词检索 | 快速原型 |
| ElasticSearch（生产） | 全文检索引擎 | 支持中文分词、布尔查询 |

### 3.3 元数据存储

- 使用 SQLite（开发）或 MySQL（生产）存储 chunk 元数据。
- 元数据与向量存储通过 `chunk_id` 关联。

## 4. 索引更新机制

### 4.1 增量更新流程

```
1. 检测变更文档（通过 content hash 比对）
2. 仅对变更文档执行：
   a. 删除旧 chunks（按 source_id 批量删除）
   b. 重新切分 → 生成 Embedding → 写入向量存储
   c. 更新元数据表（version、updated_at）
3. 记录更新日志
```

### 4.2 全量重建

- 触发条件：Embedding 模型变更、切分策略变更、索引损坏。
- 流程：新建索引 → 全量写入 → 切换别名 → 删除旧索引（蓝绿切换）。

### 4.3 并发控制

- 写入期间读取不阻塞（MVCC / 别名切换）。
- 同一 `source_id` 的更新串行化（分布式锁）。
- 批量写入使用事务保证原子性。

---

## 5. 索引配额与限制

| 指标 | 限制 | 说明 |
|------|------|------|
| 单文档最大大小 | 10 MB | 超出需拆分后入库 |
| 单 chunk 最大 token 数 | 512 | 超出截断 |
| 索引总条目上限 | 1000 万 | 超出需分片 |
| 单次批量写入上限 | 1000 条 | 防止内存溢出 |
| Embedding 调用 QPS 上限 | 100 | 按 API 限速控制 |
