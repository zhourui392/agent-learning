# W3 RAG 名词解释

## 核心概念

### RAG（Retrieval-Augmented Generation，检索增强生成）

一种让大语言模型（LLM）在回答问题前，先从外部知识库中检索相关信息，再基于检索到的内容生成答案的技术。类比：开卷考试——先翻书找答案，再用自己的话回答。

### Embedding（向量嵌入）

把一段文字转换成一组数字（向量）的过程。转换后的向量能表达文字的语义，语义相近的文字会得到相近的向量。例如"苹果手机"和"iPhone"的向量会非常接近。

### Chunk（片段/分块）

把一篇长文档切割成的小段落。因为 LLM 有输入长度限制，不能把整篇文档塞进去，所以需要先切成小块，检索时只取最相关的几块。

### Token

LLM 处理文本的最小单位。英文中大约 1 个单词 = 1-2 个 token，中文中大约 1 个字 = 1-2 个 token。LLM 的输入窗口按 token 计费和限制。

## 检索链路（四层架构）

### 召回（Retrieval / Recall）

检索的第一步：从海量文档中快速筛选出"可能相关"的候选集。追求"宁可多选，不可漏掉"，类比：先用渔网把鱼捞上来，不管大小。

### 重排（Reranking）

检索的第二步：对召回的候选结果进行精细排序，把最相关的排到前面。类比：捞上来的鱼按大小、新鲜度重新排列。

### 压缩（Compression）

检索的第三步：从排序后的结果中裁剪掉重复和低价值内容，控制送入 LLM 的文本量。类比：只挑最好的几条鱼装箱，箱子大小有限（token 预算）。

### 组装（Assembly）

检索的最后一步：把筛选后的片段拼接成一段连贯的上下文文本，供 LLM 阅读和回答。

---

## 召回层相关

### 向量召回（Vector Retrieval）

通过比较"查询文本的向量"和"文档片段的向量"之间的距离/相似度来检索。能理解语义，比如搜"怎么退货"能召回"退款流程"。

### 关键词召回（Keyword Retrieval）

通过匹配查询中的关键词和文档中的关键词来检索，类似传统搜索引擎。精确但不理解同义词。

### 双路召回（Hybrid Retrieval）

同时使用向量召回和关键词召回，然后合并两路的结果。兼顾语义理解和精确匹配的优势。

### BM25

一种经典的关键词检索评分算法。根据关键词在文档中出现的频率、文档长度等因素计算相关性分数。ElasticSearch 默认就用这个算法。

### RRF（Reciprocal Rank Fusion，倒数排名融合）

一种合并多路检索结果的方法。核心思想：如果一个文档在多路检索中都排名靠前，那它的最终排名应该更高。计算公式简单，不依赖分数的绝对值，只看排名位置。

### ANN（Approximate Nearest Neighbor，近似最近邻）

在海量向量中快速找到与查询向量最相似的 K 个向量的算法。"近似"意味着用少量精度换取巨大的速度提升（从遍历百万条变为毫秒级查询）。

### TopK / TopN

取排名前 K/N 个结果。例如 Top5 就是取前 5 个最相关的片段。

## 重排层相关

### 相关性（Relevance）

文档片段与用户查询的语义匹配程度。越相关分数越高。

### 时效性（Freshness）

文档的新鲜程度。最近更新的文档得分高，很久没更新的文档得分低。使用指数衰减函数——越旧衰减越快。

### 来源可信度（Authority）

数据来源的权威程度。官方文档 > 数据库记录 > API 数据 > 用户生成内容。

### Cross-Encoder

一种精排模型，把查询和文档一起输入模型打分。精度高但速度慢，适合对少量候选结果做精细排序（对比：向量召回是先分别编码再比较，速度快但精度略低）。

---

## 压缩层相关

### Token 预算（Token Budget）

给检索上下文分配的 token 上限。LLM 输入窗口有限（如 4096 tokens），需要给系统提示、历史对话、用户输入各留一部分，剩下的才是检索内容的配额。

### 去重（Deduplication）

移除内容高度重复的片段。两种方式：精确去重（内容完全相同）和近似去重（内容相似度超过阈值，如 85%）。

### Jaccard 相似度

衡量两个集合相似程度的指标。公式：交集大小 / 并集大小。值域 [0, 1]，1 表示完全相同。在这里用于比较两段文本的词汇重叠程度。

### 信息密度（Information Density）

有效信息在文本中的占比。如果一段文字全是标点符号或空白，信息密度就很低，应该被剔除。

## 缓存与降级相关

### LRU（Least Recently Used，最近最少使用）

一种缓存淘汰策略。当缓存满了需要腾位置时，优先淘汰最久没有被访问过的条目。

### TTL（Time To Live，存活时间）

缓存条目的有效期。超过 TTL 后自动过期失效，下次查询需重新计算。

### 缓存命中 / 缓存未命中（Cache Hit / Miss）

查询缓存时，如果找到了有效结果叫"命中"（直接返回，速度快）；没找到叫"未命中"（需要重新走完整检索流程）。

### 降级（Degradation / Fallback）

当系统部分功能异常（如检索超时、服务不可用）时，自动切换到备用方案以保证系统仍能响应。例如：向量服务挂了就只用关键词检索；检索全挂了就返回缓存中的旧结果。

## 索引与数据同步相关

### 倒排索引（Inverted Index）

一种支持关键词快速查找的数据结构。从"词"到"包含该词的文档列表"的映射。例如："退款" -> [文档1, 文档5, 文档12]。ElasticSearch 的核心数据结构。

### CDC（Change Data Capture，变更数据捕获）

实时捕获数据库中数据变更（增删改）的技术。常见工具：Debezium、Canal。用于实时同步数据库变更到搜索索引。

### 蓝绿切换（Blue-Green Deployment）

用在索引更新场景：先在新索引（绿）上构建完整数据，构建完成后把查询流量从旧索引（蓝）切到新索引，再删除旧索引。保证更新过程中查询不中断。

## 评测相关

### Recall@K（K 位召回率）

在返回的前 K 个结果中，命中了多少个期望的相关文档。公式：命中的相关文档数 / 总相关文档数。1.0 表示全部相关文档都在 TopK 中被召回。

### MRR（Mean Reciprocal Rank，平均倒数排名）

第一个正确结果出现在第几位的倒数。如果正确结果排在第 1 位，MRR=1；排在第 2 位，MRR=0.5；排在第 3 位，MRR=0.33。衡量"用户多快能看到正确答案"。

### Answer-F1（答案 F1 分数）

衡量生成答案与期望答案的词汇重叠程度。综合了精确率（生成的词有多少是对的）和召回率（期望的词有多少被生成了）。值域 [0, 1]，1 表示完美匹配。

### P95 Latency（P95 延迟）

把所有请求按延迟从小到大排序，第 95% 位置的延迟值。意思是 95% 的请求都比这个值快。比平均值更能反映"最差情况下用户要等多久"。

## 存储与工具

### FAISS

Meta（Facebook）开源的向量搜索库，支持在内存中对大规模向量做高速相似度搜索。适合开发测试，不支持持久化。

### Milvus / Qdrant

分布式向量数据库，支持持久化存储、动态增删、元数据过滤。适合生产环境的向量检索。

### ElasticSearch（ES）

分布式全文搜索引擎，支持中文分词、BM25 检索、布尔查询。在 RAG 中用作关键词召回后端。

### tiktoken

OpenAI 开源的 token 计数库，可以精确计算一段文字对应多少个 token。比估算更准确。
